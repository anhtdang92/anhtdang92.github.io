ML
kraken-ml-trading-strategy
Pythonscikit-learnGoogle CloudKraken API
ML-powered cryptocurrency trading strategy. Feature engineering on live market data, model training with scikit-learn, and cloud deployment on GCP for automated signal generation.
☁️ Cloud-deployed · Real-time API integration · End-to-end ML pipeline

Problem

Cryptocurrency markets generate massive amounts of OHLCV data but most retail trading strategies are based on lagging indicators or gut feel. I wanted to build a systematic, ML-driven approach that could identify short-term momentum signals from raw market data — tested on real Kraken exchange data.

Approach

Built a feature engineering pipeline that transforms raw candlestick data into 20+ technical features (RSI, MACD, Bollinger Band width, volume momentum, etc.). Trained gradient boosted classifiers using scikit-learn with walk-forward validation to avoid lookahead bias. Deployed the model on Google Cloud Functions to generate signals on a schedule via the Kraken REST API.

Key Decisions

Chose scikit-learn over deep learning because tabular features + gradient boosting consistently outperform neural nets on structured financial data at this scale. Used walk-forward validation instead of random splits to respect the time-series nature of the data. Deployed on GCP Cloud Functions for serverless execution rather than maintaining a VM.

Results

The model achieved a Sharpe ratio above 1.0 on backtested data with a 55%+ directional accuracy. More importantly, it demonstrated that a systematic ML pipeline — data ingestion, feature engineering, model training, deployment — can be built end-to-end by one person using open-source tools.

AI
deepseek-local-inference
PythonDeepSeek-R1RTX 4090Quantized Models
Local LLM inference and benchmarking with DeepSeek-R1-Distill-Qwen-32B on an RTX 4090. Testing quantized model performance for on-device reasoning tasks.
🖥️ 32B parameter model · Local inference · VRAM optimization on consumer GPU

Problem

Cloud LLM APIs are expensive for experimentation and raise data privacy concerns for financial services use cases. I wanted to test whether a 32B parameter reasoning model could run locally on consumer hardware with acceptable latency for real analytical tasks.

Approach

Set up DeepSeek-R1-Distill-Qwen-32B on an RTX 4090 (24GB VRAM) using quantized model formats (GGUF Q4_K_M, Q5_K_M). Benchmarked inference speed, VRAM usage, and output quality across different quantization levels and context lengths. Tested with financial reasoning tasks like interpreting credit risk metrics and generating SQL queries from natural language.

Key Decisions

Chose the 32B distilled variant over the full 671B model for VRAM feasibility. Used llama.cpp for inference instead of vLLM because it handles quantized models better on single-GPU setups. Tested multiple quantization levels to find the sweet spot between quality and speed.

Results

Q4_K_M quantization fits in 20GB VRAM and generates ~15 tokens/sec — usable for interactive tasks. The model produces surprisingly strong SQL generation and financial reasoning outputs. Proved that local inference is viable for sensitive data analysis where cloud APIs aren't an option.
